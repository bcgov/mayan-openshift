========
Security
========

This document outlines the security weaknesses found or published for
Mayan EDMS along with the information about when and in what version these
issues were fixed.


A call to ethical reporting
===========================

Security issues in Mayan EDMS are reported and addressed through direct technical collaboration with security researchers. In many instances however subsequent reporting by those researchers has inaccurately and disproportionately characterized the nature and impact of the issues involved.

Inaccurate and exaggerated claims made by security researchers regarding the scope or impact of reported issues undermine the seriousness of disclosure practices and introduce unnecessary friction between researchers and open source software maintainers. It also creates unwarranted concern among users evaluating the security posture of an open source project.

When findings are publicly described without technical substantiation or in contradiction to the actual behavior of the software, it erodes trust and risks discouraging future collaboration. This outcome is counterproductive to improving software security and places an avoidable burden on maintainers to correct the public record.


CVE-2025-14691
==============

Description: client-side navigation validation issue introduced by a regression in the JavaScript code responsible for handling fragment-based navigation when breaking out of an AJAX container.

The root cause is the insufficient restriction of URL fragments before they are interpreted by the browser as navigation targets. This issue is being reported by third parties as an

This issue does not involve the authentication system, does not bypass access controls, and does not grant unauthenticated users any form of access to Mayan EDMS data or functionality. No server-side logic, permission checks, document access paths, or API endpoints are affected by this CVE.

Exploitation if any is limited strictly to the user's own browser session and requires that the user voluntarily load a crafted URL which normally fails the most basic visual inspection. The behavior executes entirely within the client browser and does not allow access to documents, metadata, or configuration data beyond what the user is already authorized to view. There is no mechanism, either actual or theoretical, by which this issue could be used to steal documents, impersonate other users, or access the system without authentication. Claims stating otherwise are incorrect and conflate client-side navigation behavior with server-side authorization, which remains fully enforced.

Impact: Negligible

Affected Version: Between 4.7 and 4.10.1

Fixed Version: 4.10.2, 4.9.7, 4.8.10, 4.7.8, 4.6.12.


CVE-2025-14692
==============

Description: client-side open redirect condition, also caused by the same regression in fragment-based navigation handling.

Under specific circumstances, an unvalidated fragment could cause the browser to navigate away from the Mayan EDMS interface. This behavior is purely client-side and does not provide access to internal routes, documents, or APIs, nor does it weaken authentication or session handling in any way. The server does not process, trust, or act upon the redirected location.

As with CVE-2025-14691, exploitation if any is limited to the local browser session of the user who voluntarily and knowingly follows a crafted link which visually contains malicious and abnormal content in the URL. It cannot be used to exfiltrate data, escalate privileges, or gain unauthorized access to the application. There are no known or plausible attack chains that combine this behavior with other components of Mayan EDMS to compromise confidentiality, integrity, or availability. Assertions that this issue allows unauthenticated document access or system compromise are factually incorrect.

Impact: Negligible

Affected Version: Between 4.7 and 4.10.1

Fixed Version: 4.10.2, 4.9.7, 4.8.10, 4.7.8, 4.6.12.


CVE-2022-47419
==============

Description: The select2 template used to display tags in the attach and
remove view did not performed complete tag label sanitation and allowed
cross-site scripting (XSS) to occur.

This is a limited scope weakness and can only be used at best to perform
phishing attacks. However this action requires for an attacker to have
a user account and be a trusted user with the necessary permissions to create
or edit tags.

Any usage of this weakness remains logged in the event system making
it easy to track down any bad actors.

Visitors to a Mayan EDMS installation cannot exploit this weakness.

It is not possible to circumvent Mayan EDMS access control system or
expose arbitrary information with this weakness.

Contrary to some incorrect reports, it is not possible to exploit this
weakness to steal the session cookie and impersonate users. Since version 1.4
(March 23, 2012) Django has included the ``httponly`` attribute for the
session cookie. This means that the session cookie data, including
``sessionid``, is no longer accessible from JavaScript.
https://docs.djangoproject.com/en/4.1/releases/1.4/

Mayan EDMS currently uses Django 3.2. Under this version of Django
The ``SESSION_COOKIE_HTTPONLY`` defaults to ``True``, which enables the
``httponly`` for the session cookie making it inaccessible to JavaScript
and therefore not available for impersonation via session hijacking.
https://docs.djangoproject.com/en/3.2/ref/settings/#session-cookie-httponly

Django's ``SESSION_COOKIE_HTTPONLY`` setting is not currently exposed by
Mayan EDMS' setting system, therefore it is not possible to disable this
protection by conventional means.

Due to all these factors, the surface of attack of this weakness is
very limited, if any.

There are no known actual or theoretical attacks exploiting this
weakness to expose or destroy data.

Impact: Low

Affected Version: 4.4.2 and earlier

Fixed Version: 4.4.3 (2023-02-11), 4.3.6 (2023-02-19), 4.2.14 (2023-03-09), 4.1.11 (2023-03-08)

Security FAQ
============


I found a security issue
------------------------

Open an issue at |SOURCE_CODE_ISSUES| and mark it as
Confidential. Allow us at least 48 hours to find and release a fix
for the issue before submitting it to the CVE database.

The issue must include a reproducible proof-of-concept exploit.

The exploit must be shown to happen in a production deployment using a non
self-signed SSL certificate.

Exploits that are only reproducible on a development or testing deployment
using Django's insecure `runserver` (https://docs.djangoproject.com/en/4.2/ref/django-admin/#runserver)
are not valid.

Theoretical vulnerabilities, speculative risks, or static analysis output
without real-world impact are not security issues.

Likewise reports consisting of configuration recommendations or generic
best-practice checklists will not be treated as security vulnerabilities.

For verification, all security reports must be digitally signed with a
valid GPG key to ensure the authenticity and integrity of the
communication.

Disclose any relevant affiliation and any conflicts of interest.

We maintain a strict policy against unethical behavior in the
vulnerability disclosure process. Attempts to submit fraudulent,
AI-generated, or non-actionable CVEs will be formally reported as
abuse.

We reserve the right to request blacklisting for
individuals or organizations that misuse either the CVE or our disclosure
process.

I want to contact the team directly to report a security issue
--------------------------------------------------------------

The contact method is via email info@mayan-edms.com.


Are security bounties available?
--------------------------------

As a free and open source project, bounties or prices for security
issues are not available.

Blackmail, extortion, or paid registrations as requirement for security
issue disclosure will be reported to the pertinent managing or legal
authorities.
